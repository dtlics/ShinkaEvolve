# Audit Remediation Report

## 1. Coverage verdict

**No — we did not address every flaw, and we should not pretend otherwise.** The audit's load-bearing findings (the ones that determined whether a real evolutionary run is healthy or quietly broken) are fixed and test-guarded. A tail of lower-severity items — mostly dormant code paths, asymmetries that only bite under non-default knobs, and documentation drift — is still open and named explicitly below.

Tally across all 73 findings:

| Verdict | Count |
|---|---|
| **Addressed** | 41 |
| **Addressed as a mutable lever** (uncertain call exposed as a knob) | 3 |
| **Partial** (core fixed, a piece of the promised work missing) | 16 |
| **Deferred — intentional** (latent on a path we don't use) | 3 |
| **Not addressed** | 10 |
| **Refuted** (audit was wrong) | 0 |

**The partials and not-addressed items, stated plainly and up front:**

- **Repair prompts are still blind for the most common cnot failure.** When the outer loop samples a previously-broken candidate to repair, a crash or timeout now carries its error text into the repair prompt — but a *domain* failure (the dominant "non-adjacent gate" class that the truthfulness fix just made visible) carries no traceback and its rich reason is not persisted along that path, so those repairs still read "no error output." The immediate-fix path *does* get the rich reason; the sampled-parent path does not yet. **(partial)**
- **The graceful wallclock guard is still dead under shipped defaults.** The two timeout constants are now overridable, but every shipped config still pins the per-eval kill at a value that violates the required ordering, and the promised config-lint test that would catch this was not shipped. One pathological trial can still eat a whole window unless an operator overrides the default. **(partial)**
- **The cost bandit's exploration bonus still shrinks on a rejected slot.** Three of the four reasons a good-but-pricey model could never recover are fixed; the fourth is not — a novelty-rejected attempt still counts as "tried" without ever counting as "completed," so the arm's exploration credit keeps decaying on rejects. **(partial)**
- **The rewrite safety-net's smoke example still teaches a deprecated guard**, even though the playbook and the real decision function are fixed. And the mutation step itself, though now a protected target, still has no real output-contract check — a rewrite that drops its required outputs would pass validation. **(partial, ×2)**
- **The budget overshoot bound is documented and loosely fenced but not proven by a test**, and inside an already-admitted slot the first mutation and embedding still fire without a re-check. **(partial)**
- **A cluster of documentation/sensor inaccuracies is untouched**: a sensor's own contract docstring omits ~10 fields it actually emits; a few diagnostics keys are emitted but undocumented; one stale trigger-metric default is still threaded through; a rollback example in the playbook still omits two required arguments and would raise on a literal copy; the repo's script index and an architecture doc still reference removed files; and several test-coverage gaps (the weights peek, weighted meta-direction sampling, an iteration counter, the bundle validator, a double-count negative test) remain unguarded. **(not addressed / partial)**
- **Latent-but-dormant, left intentionally**: scheduled migration stamps the wrong generation, an embedding-cost asymmetry can make a duplicate-prone arm look cheap, a post-eviction native sampler regression, and the no-model-pool bandit no-op. Each only fires on a configuration or code path the framework does not ship or the orchestrator does not use.

None of the open items reintroduces a silent correctness break on the default path. The ones that mattered most for a truthful, self-correcting run all landed.

## 2. The narrative

### Truthful fitness — the foundation everything else stands on

The worst flaw in the whole audit lived here. The shipped task evaluator reported *every* crash, timeout, and invalid circuit as a correct program that simply scored zero. That single mislabel poisoned three systems at once: the repair loop never fired (there was nothing marked broken to repair), the failure-rate sensor read healthy while garbage flowed into the archive, and the selection bandit learned from that garbage — so the search was quietly optimizing for "cheap and not-NaN" instead of "actually better." A whole pillar of the design (evaluate, detect failure, repair) was dead on the real task, and the reference task masked it because *it* signalled incorrectness correctly.

The fix makes the evaluator tell the truth: an invalid or crashing candidate is now reported as incorrect, success is reported as correct, and the shared wrapper honors that signal end-to-end. Crucially, a valid-but-no-improvement candidate is now held strictly separable from an invalid one in both reward and parent selection, so "didn't help" and "is broken" never collapse into the same bucket again. With a real failure signal, the repair loop engages, the health sensor reflects reality, and selection pressure finally points at genuine improvement. The rich failure reason ("non-adjacent two-qubit gate") now also rides into the immediate-repair prompt, and the failure-feedback channel defaults on, so the model is told *why* it failed rather than handed a blank.

Two pieces here are still partial and worth the colleague's attention. First, the *sampled-parent* repair path — when the outer loop deliberately picks an old broken candidate to fix — still only forwards a crash traceback, and the dominant domain-failure class produces no traceback, so those particular repairs remain blind. The immediate-fix path is fine; this is the deferred second consumer. Second, the evaluator's graceful self-abort budget is still unreachable because every shipped config pins the hard kill too low, and the lint test that would have caught the bad config never shipped. Both are real, both are scoped, neither silently corrupts the archive — they degrade the *quality* of recovery, not the truthfulness of the signal.

### Islands — making differentiation real instead of cosmetic

The point of islands is to explore different directions in parallel and migrate the winners. The audit found the differentiation was largely theatre: the per-island "direction" lever was read but nothing ever wrote one, the policy file that was supposed to decide spawns and migrations was never actually called, the two island-selection knobs were hardcoded away, and the diversity sensor reported raw member count — so two islands that had collapsed onto the same genome looked maximally diverse.

All of the load-bearing pieces are now genuinely wired. A per-island direction can be authored and is fed into that island's prompts, so islands can pull apart. The policy's spawn/migrate decisions are now executed at the window boundary under an opt-in flag, so editing the policy actually changes behavior. The selection knobs (how parents are drawn, whether inspiration crosses island boundaries) are honored with defaults that reproduce prior behavior. The diversity sensor now measures real spread — mean pairwise distance in embedding space — plus a true "generations since this island improved" stagnation count, so a collapse reads as a collapse. A persistent "known failure modes to avoid" caution is now rendered as its own always-on prompt section, so swapping an island's direction can never silently drop that caution. And the cap on island count is now respected by both spawn paths, with an explicit warning when it has to soft-overshoot because everything is protected.

Two small items remain. The stagnation-triggered spawn still clones a source program without an automatically paired fresh direction — mitigated because that path is dormant by default and an operator can now author the direction by hand. And scheduled migration still stamps the run's max generation rather than the triggering program's — latent because the migration rate ships at zero. Both are dormant-path issues, not active regressions.

### Selection & recovery — letting a good-but-expensive model come back

This cluster is about the outer loop's ability to *recover* a search that has narrowed onto the wrong arm. The audit found a bandit that, on a hard task, could never rescue a good-but-pricey model: correct-but-worse children scored identically to outright failures, the cheapest arm won purely on cost, and a duplicate-prone arm looked even cheaper and less-tried than it really was. Worse, the actual recovery levers — the exploration floor, the force-explore override, the arm-subset restriction — were either undocumented or required a code rewrite to reach, so an agent facing a collapsed bandit reached for surgery instead of a switch. And a reproducibility seed was being pinned identically every generation, which collapsed operator diversity to a single patch type.

Most of this is fixed and, importantly, *taught*. A correct candidate now scores strictly above a failed one, so "worse but valid" can still earn exploration. A novelty-rejected attempt now bills its real spend to the arm. The cost sensitivity was lowered in the shipped scaffolds and documented as a knob. The seed is now offset per generation, so the operator mix and the forced-exploration draw vary as intended. And the three recovery levers are now plain config flips, documented with explicit "flip this first, before any rewrite" guidance.

Two gaps remain. The fourth driver of bandit collapse is unfixed: a rejected slot still increments the arm's "tried" count without a matching "completed," so the exploration bonus keeps deflating on rejects — the exact failure mode the reward-floor fix was meant to end, still reachable through the visit-count side door. And once a recovery lever *is* engaged, the read-only diagnostics peek still reports the full-pool learned posterior rather than the distribution the run is actually using, so the operator's view of "what is the search doing now" is stale precisely when they've intervened. A legacy non-Azure web-search path that silently drops the request is now documented but still un-warned — dormant in the Azure-only deployment we ship.

### Budget — making spend honest and the hard-stop trustworthy

The budget hard-stop only matters if the ledger is honest and the expensive discretionary calls respect it. The audit found three holes. A capped or failed Azure call threw away already-billed partial output and recorded zero, and the meta round had no error handling so the same failure aborted the whole cycle. The playbook contradicted itself on cost logging, *guaranteeing* every expensive deep-research call was counted twice and tripped the stop early. And the two priciest discretionary calls fired unconditionally with no pre-flight, so invoking them with too little headroom blew the cap by their full cost.

These are fixed. Billed partial output is now returned on a cap-hit and the billed cost is attached on a true failure, so the ledger reflects real spend and meta degrades gracefully instead of crashing. The double-count is gone: the guidance now uniformly says "let the call self-log once, never log it twice." Both expensive calls now run a pre-flight and return a no-spend "skipped" result when the remaining budget can't cover the estimate. And a billed call that can't be priced (a renamed or typo'd deployment) now emits a loud warning instead of silently booking zero, so an unpriced path is visible rather than draining quietly.

The partial here is honesty about the overshoot *bound*: the true bound ("at most one full slot — a mutation plus its fix-retries plus embeddings") is now documented and the fix loop re-checks before each retry, but there's no test asserting the upper bound, and within an already-admitted slot the first mutation and embedding still fire without a re-check. Two minor asymmetries are left intentionally: embedding spend is counted against an arm on a reject but not on an accept (a duplicate-prone arm can look slightly cheap), and a bare single-model pin skips the bandit entirely (correct — there's nothing to rank). The deep-research web-search surcharge was turned into a documented, conservative, overridable knob rather than a buried constant.

### Safety net — so a bad rewrite can't corrupt the thing that rewrites

The outer loop is allowed to rewrite its own strategy code, which means the deploy/rollback machinery is load-bearing for safety. The audit found it dangerously permissive: snapshot/deploy/rollback would write to *any* target name, so an off-by-one could overwrite a foundation file — the JSON contract, the evaluator, the diagnostics — and even make the corruption the restore point. Multi-file "concern bundles" were applied one file at a time with the index and rollback handle written only at the end, so a mid-bundle failure left the workspace half-overwritten with no way back — the exact incompatible state bundles exist to prevent. The validator green-lit any file containing a main function regardless of its name, and one structured-output contract checked only a legacy key, so a rewrite that dropped the consumed fields would pass validation and then silently break the run. And the rollback decision function — the safety net itself — had no test, was blind to the owner's flagship reward/bandit-regression trigger, and treated missing novelty data as maximal diversity.

The core is now solid. A mutability gate rejects any write to a non-mutable target (with an explicit tooling override), guarding the foundation files. Bundles now snapshot everything first and, on any failure mid-apply, restore every touched file and re-raise *without* recording the deploy — leaving the workspace byte-identical to before. The validator now rejects unknown or typo'd target names instead of waving them through, and the structured-output contract now requires all the consumed fields. A re-deploy of an already-rejected strategy is now blocked unless explicitly forced. And the rollback decision function has a real test exercising bandit-collapse, a stable-hard-task no-rollback, and near-total collapse; it now catches single-arm collapse independent of the improvement delta, treats absent novelty as unknown rather than diverse, and has an absolute correctness floor for an all-broken rewrite — and the playbook teaches it over the deprecated guard.

Two partials. The standalone smoke example still demonstrates the deprecated guard even though the playbook and the real function are fixed — a teaching inconsistency, not a behavior bug. And the mutation step, now a protected target, still lacks a real output-contract check, so a rewrite that dropped its required outputs would pass parse-only validation; the unknown-target leak around it is closed, but the positive contract is still owed. A single-file rollback still doesn't pre-snapshot the bytes it discards — left as-is and safe, because every deploy already snapshots the live bytes the moment they go live, so nothing is unrecoverable.

### Meta-loop — keeping the recurring caution and the direction in the prompt

The meta loop's job is to look across the recent population and steer the next windows: here's a direction, and here are the failure modes to avoid. The audit found the caution kept getting dropped. On crossover and literature-grounded generations the prompt lost *both* the direction and the failure caution; whenever the sampled direction list came back empty the caution vanished too; repair prompts got neither, and could happily re-introduce the exact failure class meta had just warned about. On top of that, meta's failure classifier kept only the first traceback line — the generic "Traceback (most recent call last):" banner — stripping the actual exception type, so meta saw a useless banner for the entire correctness-failure class. And meta started each round with no memory of its own prior outputs, so it re-spent budget re-proposing directions it had already given.

This cluster is fully addressed. The failure caution is now an always-on prompt section, decoupled from the direction — it renders on every patch type, even when no direction was sampled, and now reaches the repair prompts too, so a fix can't blindly re-introduce a warned-against failure. The classifier now grabs the *last* meaningful traceback line when the first is the standard banner, so meta sees the real exception. And meta now auto-fills its "what I already suggested" memory from recent rounds, so it stops paying to repeat itself. The one judgment call — how to split meta's attention between failures and top performers — was turned into a documented knob defaulting to the old even split, with explicit guidance to bias toward failures in a failure-dominated window.

### Sensors & contract — the dashboards the outer loop steers by

The outer loop can only act on what the sensors report, so a dark or mislabeled sensor is a real hazard. The headline finding: the diagnostics field that the debug-agent escalation keys on was hardcoded to empty, so the "a candidate exhausted its retry budget" rung could never fire from real data. Two more mattered: after a rewrite the low-window streak was inherited unchanged, so a single bad window could immediately yank control back before the new strategy got a fair trial; and the convert scaffold still shipped the old fixed stagnation trigger that flags tiny-but-real gains as stagnation — the very false-stagnation bug the hybrid trigger was built to kill.

All three are fixed. A slot that's still incorrect after the repair loop now records its real identifier into the exhausted-retry list, so that escalation rung can fire on real data. A rewrite now zeroes the carried low-streak when the strategy fingerprint changes, giving every new strategy a clean fair-trial window. And the convert scaffold now ships the hybrid floor trigger with no stale key, matching the primary scaffold.

What's left here is documentation and edge-case drift, not function. The sensor's own contract docstring still describes a minimal field set and omits roughly ten fields it actually emits; a couple of emitted keys are undocumented; one trigger-metric parameter is still threaded with a stale default and then dropped; one metric is mislabeled in the concern map as a window-level signal when it's really per-program; an input flag the playbook leans on is missing from a script's documented inputs; and a zero-window invocation returns a near-empty diagnostics dict. None of these corrupts a run, but each is a genuine inaccuracy an operator could trip over, and they're called out rather than papered over.

### Orchestrator docs — the playbook the agent actually follows

The documentation *is* the interface the outer-loop agent acts through, so a contradiction in the playbook is a real bug. The audit found several. The agent's headline duty — noticing a model the bandit has starved due to a *selection* flaw, independent of any stagnation symptom — appeared nowhere as a standalone check; it was only reachable as stagnation-ladder rungs, so a healthy rising run with a locked-out arm would never trigger it. The budget-logging contradiction (covered above) lived here too. The main-loop section told the agent to step one window per turn while the cost sections demanded run-to-decision — and stepping one window per turn pays an agent turn per window, eroding the whole cost asymmetry the framework is built on. A subagent was told the inner loop had "already retried three times" when the default is one retry, biasing it toward over-diagnosing. And the deep-research field that triggers content-filter refusals was misattributed to the agent's own query authoring when it actually lives in an immutable prompt, leaving the agent with no real remedy.

Most of this is fixed. There's now a dedicated "is this arm locked out or truly bad?" section that runs even on a healthy rising run, distinguishes a starved arm (near-zero selection but positive reward) from a genuinely weak one, and prescribes a config flip before any rewrite. The main loop now teaches run-to-decision as the normal mode and explicitly demotes single-stepping to debugging. The subagent now states the true retry budget. The deep-research guidance now correctly tells the agent the offending field isn't theirs to set, that the only lever is query shape, and to rewrite the query (never retry) on a refusal — plus a mandatory pre-flight self-check. And there's a new "flip a knob before you rewrite code" catalogue of ~18 mutable levers with defaults and when-to-use guidance — the single most important cultural fix, because it routes the agent toward cheap reversible config changes before expensive irreversible rewrites.

Three doc items are still open. An architecture doc and the repo's script index still reference removed files (the doc now carries a "historical" banner and the superseded formula is marked deprecated, but the dead references persist). A rollback example in the playbook still omits two required arguments, so a literal copy would raise. And one consistency wart runs through the otherwise-good levers catalogue: the cost-sensitivity knob is documented with one default in the table and a different one in the example config and code defaults. These are accuracy fixes still owed, not behavioral regressions.

### Tests — pinning the fixes so a refactor can't quietly undo them

The whole point of the test work was to make sure the fixes above can't be silently reverted with CI staying green. A noisy class of warnings (tests returning truthy values, which a future test runner will treat as a hard error and which can mask an assertion-free test) is fully cleaned up, and the suite passes offline with that warning promoted to an error. The large majority of the determinate new fixes now have targeted guards: the truthful-evaluator behavior, the billed-partial-output handling, the reward floor, the rollback basket and foundation-write guard, the always-on failure caution, the island brief round-trip, and the island selection/diversity/capped-spawn behavior.

The honest gaps: the bandit parity test was upgraded to compare against a live reference (so it now catches wrapper/state drift) but its third arm still gets no updates, so the reward-ranking path it was meant to exercise still isn't actually tested. The weights peek has no test caller at all. The budget overshoot upper bound is asserted only as a lower bound. And four smaller pins are still missing — weighted meta-direction sampling (tested only with a degenerate weight that can't distinguish weighted from argmax), an iteration counter that's never asserted, the bundle validator that no test invokes, and a negative test proving the double-count is avoided. Nothing in this cluster is fully untested; the holes are specific and named.

## 3. Closing: "not perfect yet" is the intended state

The owner's design principle is the throughline of this whole remediation: the framework is a cheap, high-volume inner loop wrapped in a smart outer loop that is allowed to rewrite its own strategy and recover — and every genuinely *uncertain* decision was deliberately turned into a runtime lever the outer-loop agent can change, then documented so the agent reaches for the lever before it reaches for surgery. The cost sensitivity, the exploration floor, the force-explore override, the failure-vs-performer split, the validity floors, the web-search surcharge, the brief-compose mode, the partial-output handling — none of these were frozen into a guess. They were exposed as knobs with defaults that reproduce prior behavior and taught with explicit "flip this first" guidance. That is exactly why a residual list of partials is acceptable rather than alarming: the items that would silently corrupt a run (truthful fitness, the foundation-write guard, atomic bundles, the rollback decision, the budget ledger) are closed and pinned, and most of what remains is either a dormant path or a tuning question the operator can now answer at runtime without touching code. "Not perfect yet" isn't a gap in the plan — it's the plan working as intended.
