## 1. Guiding principle & how this plan honors it

**The owner's central rule:** the end state does not have to be perfect. For any design decision we are unsure how to make right now, the fix counts as **SUCCESSFUL** if we (a) leave that decision as a **CLEARLY MUTABLE lever** the runtime orchestrator can change, and (b) **teach the orchestrator** (in the docs) that the lever exists and its trigger signal. Converting an uncertain hardcoded choice into an exposed, documented, runtime-tunable lever **is the win.**

Every fix below is classified as exactly one of:

- **DETERMINATE-FIX** — a clear bug with an obviously-correct change (a dropped cost, a seed not offset, a self-contradictory doc, dead code with one right wiring). The plan specifies the correct change.
- **MUTABLE-LEVER** — the right policy is genuinely uncertain. We do **not** guess a perfect value; we expose a config knob (`evo.*` / `db_config.*`) or a mutable strategy file behavior, pick a **conservative default that preserves today's behavior**, and specify the doc update that teaches the lever + its trigger.
- **HYBRID** (the common case) — a determinate mechanism fix that **also** creates a mutable lever. The mechanism that *signals* something must be fixed determinately; the *shape* of the policy becomes a knob.

**Two structural preferences enforced throughout:**

1. **MUTABLE-over-FOUNDATION.** Whenever a problem can be solved in a `scripts/*.py` strategy file or `evo.*`/`db_config.*` config **or** by editing a FOUNDATION file, we choose the mutable path. It serves the owner criterion directly and keeps the contract stable. (Example: the entire Selection cluster is solved without touching `shinka/llm/prioritization.py` — the bandit's pickle-state contract is FOUNDATION; all its levers are reachable via `compute_reward.py`, `select_llm.py`, and `evo.*`.)

2. **FOUNDATION changes are flagged for owner sign-off.** The repo's FOUNDATION set (sqlite schema; the JSON stdin/stdout contract in `scripts/_common.py`; `tasks/*/evaluate.py` + `initial.*`; `shinka/core/wrap_eval.py`; `scripts/evaluate.py`, `archive_record.py`, `archive_query.py`, `diagnostics.py`; `harness/journal.py` + **all of `harness/*`** including `run_window.py`; `scripts/_azure.py`; `scripts/deep_research.py`) must never change mid-run; only a human changes them between runs. We (owner + this planning pass) **are** that human, so foundation changes are in scope — but each is flagged **`foundation (owner sign-off)`** with rationale + blast radius (§5).

## 2. Phased roadmap

Ordering is by **real dependency**: contract/foundation decisions first (they change what everything downstream consumes), then producer+consumer "concern bundles" together, then docs that describe shipped behavior, then the test latch.

## 8. Risks & sequencing notes

1. **`run_window.py` is the highest-contention file** — touched by Phases 1 (M5, M6, M9, validity plumb), 3 (C0, C3, C4, C6), 2 (2A.2, 2B.1), 7 (G1, G2, G4).
