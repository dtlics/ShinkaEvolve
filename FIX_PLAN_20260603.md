## Group 1 — Revert safety (C1 critical, H10 high)

### C1 — a "revert" does not rewind the strategy code
**Behavioral problem.** SKILL step 6 tells the orchestrator that `restore_state(results_dir, snap_id)` is "a FULL rewind of **code** + DB + bandit." It is not: `snapshot_state` copies only `programs.sqlite` + `bandit_state.pkl` + `journal/run.json` (`strategy_store.py:242`), and `restore_state` restores only those three (`strategy_store.py:271-279`). The strategy `.py` on disk is untouched. The per-target code rewind exists as a *separate* API, `rollback(target, prior_hash)` (`strategy_store.py:194`), which the documented flow never calls (the only callers of `rollback`/`restore_state` are the tests). Net: an orchestrator that deploys a regressing rewrite and follows step 6 rewinds the archive+bandit but leaves the **bad policy live**; the next subprocess imports it fresh and every later window runs it against a rewound archive — and the fingerprint re-stamp even grants it a fresh fair-trial streak.

**Fix (recommended: option A — make the revert atomic).** Have `deploy`/`deploy_bundle` record the pre-deploy code hashes (`prior_hash`/`prior_hashes`, already computed at `strategy_store.py:171`/`415`) into the state snapshot's `state_meta.json` as `prior_code: {target: hash}`. Then `restore_state` reads `prior_code` and copies `strategy_history/<hash>/<target>` back over `scripts/<target>` for each target (reuse the exact copy in `rollback`). One call now does what the docs already promise. The code snapshots already exist on disk, so this persists only the *mapping*. Resume-safe: the bytes hit disk before the next `run_window` subprocess imports `scripts/` fresh.
- Alternative (option B): keep `restore_state` state-only and require the agent/harness to also call `rollback_bundle`, then fix docs. Rejected as primary because it preserves the two-call/two-handle coupling that caused C1 (an improvised revert can omit one call). The tests' canonical revert uses `rollback`, proving B works — but A removes the foot-gun.
- **Files:** `strategy_store.py` **[FND]** (`snapshot_state` 227-250, `restore_state` 253-291, `deploy` 135-191, `deploy_bundle` 381-443).
- **Docs:** SKILL.md step 6 (399-401) + step 4 (384-387, "snapshots the framework files" is false today); CLAUDE.md:45; NOTES.md P9-T0 row (35); the `strategy_store.py` header (213-217) + `restore_state` docstring.
- **Tests:** extend `smoke_test.py`/`test_improvements.py` to assert `restore_state` rewinds `scripts/<target>.py` too (currently `test_improvements.py:626` scopes it archive+bandit only).

### H10 — `restore_state` rewinds the ledger when live `run.json` is corrupt at revert
**Behavioral problem.** `restore_state` preserves the live `total_cost` only when it can read it (`strategy_store.py:264-290`); on a missing/corrupt live `run.json`, `live_total=None`, the snapshot's (lower) `run.json` is copied in (271-279), and the re-stamp is skipped. The on-disk file is now a *valid* dict with a `total_cost`, so `journal.read_run`'s recompute-from-streams net (`journal.py:317-333`) does not engage — the budget railguard under-counts by (live − snapshot) at exactly the conjunction (a revert + an unreadable run.json) the durable ledger exists to defend.
**Fix.** When `live_total is None`, recompute from the durable streams (reuse `journal`'s existing recompute / `_reconstruct_run`) and re-stamp `max(recomputed, snapshot)`. The ledger can then never move down on a revert.
- **Files:** `strategy_store.py` **[FND]** (264-290), importing the recompute helper from `journal.py` **[FND]**.
- **Tests:** corrupt live `run.json` → revert → assert ledger ≥ true spend.

## Group 2 — Novelty "keep-the-better" (H5 high; closes M9/M10, touches M11/M12)

**Behavioral problem.** Novelty runs **pre-eval** (`run_window.py:689-728`, returns/drops at 728 before `_evaluate_candidate` at 730), so the newcomer has only an embedding, never a score. A near-duplicate (`max_sim ≥ threshold`) is dropped **unevaluated**, and the incumbent (possibly worse) is unconditionally kept. The user's requirement — compare the two evaluator scores and keep the better — is impossible on this path. shinka *does* ship keep-better niching (`dbase._update_archive_crowding` 2493-2554) but it's dead (`archive_selection_strategy` defaults `"fitness"`, never set to `"crowding"`), archive-full-gated, global (not island-scoped), and doesn't drop the worse newcomer from the sampling pool — so it is not reusable as-is (M9).

**Fix — split novelty into a pre-eval flag + a post-eval resolve.**
1. Pre-eval step keeps the embed+similarity probe but **does not drop**; it records `near_dup` + `most_similar_id` + (new) `most_similar_score`.
2. The near-dup candidate is **evaluated** (we need its score).
3. New post-eval "resolve": compare candidate vs incumbent **within the same island** (intra-island niching — comparing/evicting cross-island would defeat the island design): newcomer better → archive newcomer **and tombstone the incumbent** (`db.tombstone_program`, the existing non-destructive evict at `dbase.py:862`, already wrapped by `repair_record.py`); newcomer worse/equal → drop newcomer, bill the arm **cost-only** (mirror `run_window.py:718-725`); newcomer incorrect → existing eval-fail path.
- **Cost note (own it):** we now evaluate near-dups we used to drop free — inherent to "keep the better." Acceptable per the requirement; document that `code_embed_sim_threshold` now gates *eval-then-compare*, not *free drop*.
- **Counters:** re-thread `novelty_acceptance_rate` so "kept-better-evicted", "dropped-worse", and "accepted-novel" are honest (fixes the M11/M12 mis-counts, e.g. the unconditional `novelty_accepts` at `run_window.py:777`).
- **Files:** `novelty_check.py` **[MUT]** (add `most_similar_score`); `run_window.py` **[FND]** (defer the decision to post-eval, wire resolve/evict/cost-feed); a thin `archive_evict.py` wrapper over `tombstone_program` (or extend `repair_record.py`) **[FND-plumbing]**. No `dbase.py` edit (reuse `tombstone_program`; do **not** enable `crowding`).
- **Docs:** SKILL novelty rows (208-210, 491, 554) + the `code_embed_sim_threshold` lever; taxonomy novelty entry; mark audit M9/M10 resolved.
- **Tests:** near-dup-better → incumbent tombstoned, newcomer archived; near-dup-worse → newcomer dropped, cost-only; confirm tombstoned incumbents are excluded from the comparison set.

## Group 3 — Island diversity (H1/H2 high; the core fix)

**Behavioral problem (framed as a run sees it).** Today islands differ in only two cosmetic ways: (1) which `island_idx` the parent is drawn from, and (2) a per-island **brief string** swapped into the system prompt (`shinka/core/sampler.py:96-97`). The *inspiration exemplars* shown to the model are chosen purely by **score** (top-k within the island, `sample_parent.py:248-254`) — identical in kind across islands. Worse, three mechanisms compound into single-lineage convergence:
- **Global archive eviction.** The archive cap is **global** (40, `dbase.py:2399-2402`); eviction removes the *globally* worst correct program (`_update_archive_fitness` 2420-2491, no `island_idx` filter). A young/weak island's members are the global-worst, so they are evicted first → that island's archived-correct count trends to **zero**.
- **Empty-pool separation fallback.** When an island's archived pool empties, `sample_parent.py:224-230` **silently falls back to the global pool** — so that "island" now draws its parent *and* its score-ranked inspirations from the **dominant lineage**. It is now evolving the winner's descendants under its own brief text.
- **Zero migration.** `migration_rate=0.0` (`dbase.py:65`) → `ElitistMigrationStrategy.perform_migration` early-returns (`islands.py:249-250`); nothing ever moves, so there is no counter-flow.
- **Why `island_health` still says "differentiated".** The orchestrator-facing signal of differentiation is the **distinct brief strings** (meta writes one per island every window) plus per-island embedding *diversity*; no metric compares **lineage across islands**, so the briefs stay distinct (green) while the genetic material has merged. The differentiation is an illusion of text.

**Fix 3a — direction-oriented inspiration sampler [MUT].** It is fine for all islands to share the *same sampler logic*; the logic must become direction-driven. In `sample_parent.py`: read the parent island's latest brief (`db.get_latest_meta_brief(island_idx)`, already available, no payload/contract change), parse `structured_json` → `directions:[{text, weight, assigned_program_ids}]`, **sample ONE direction** (seeded RNG, for parity), and return the programs assigned to that direction as the inspirations; if a direction has no assigned/surviving programs, **fall back to the direction text only** (empty inspiration ids; `construct_mutation_prompt` already renders the text). This finally makes the brief change *which code is shown*, not just the prompt text (closes H1). Output keys unchanged (`archive_inspiration_ids`/`top_k_inspiration_ids`), so `run_window`/`construct_mutation_prompt` need no change.
- Depends on Fix 3d (meta producing the mapping).

**Fix 3b — island-aware archive capacity/eviction [FND].** Replace the single global cap with **proportional-with-floor** (recommended): every initialized island gets a guaranteed floor (e.g. `archive_floor_per_island ≈ 3–5`), the remaining slots distributed globally by fitness. Eviction picks the victim honoring per-island budgets + protections: never evict an island's **only** member, never its **best**, never the **global best**, and keep the existing **repair-tombstone-first** reclaim (2442-2454). Cleanest implementation: a `_pick_archive_victim(program)` helper used by both `_update_archive_fitness`/`_update_archive_crowding`. New `DatabaseConfig` fields (no schema change). (Alternative: strict equal per-island cap `archive_size // num_islands` — simpler, less meritocratic.)

**Fix 3c — small default migration [FND / per-run].** Turn on a small default: `migration_rate ≈ 0.05` (≥1 elite/island via `max(1,int(size*rate))`), `migration_interval` 10 (unchanged); the existing `ElitistMigrationStrategy` already does elitist random source→dest and protects each island's best — no executor change, it just needs the rate > 0. Land **with** 3b (migration alone is weak while eviction is global). Can be a per-run `db_config` value rather than editing the engine default, if preferred.

**Fix 3d — meta one-shot, per-island, code-grounded, with direction→program assignment [MUT meta + FND payload].** Today meta flattens the archive (no per-island separation) and is **blind to source code** (best omitted on the auto path `run_window.py:1092-1105`; recents gathered code-less `meta_summarize.py:144-145`) — H11. Redesign:
- Build the meta context as **clearly delimited per-island blocks**, each with that island's top-N programs (code-capped) + recent failures, plus one shared global-best anchor. Add `best_program` + per-island gather (with `include_code`, bounded `code_preview_chars`) to the meta payload.
- **Output schema:** `{islands:[{island_idx, directions:[{text, weight, assigned_program_id?}]}], global_directions:[…], failure_note}`. `assigned_program_id` present ⇒ a **realized** direction labeled onto an existing entry (the mapping 3a consumes; closes M13); absent ⇒ **net-new creative** direction. Validate assigned ids against the island's shown set in `_parse_meta` (drop unknowns; add the count/coverage/distinctness checks M14 wants).
- Persist the per-island `directions` (incl. assignments) into `meta_briefs.structured_json` via `island_brief.py`; keep `content` = headline text for back-compat.
- **Cost control:** cap top-N/island and code chars as new mutable knobs (`meta_top_n_per_island`, `meta_code_preview_chars`); raise the M4 self-skip estimate (`meta_estimated_cost_usd`) from $1.0 toward the realistic per-model worst case.

## Group 4 — Termination (H6/H7/H8 high) — user's simplified rule

**User's intended rule.** Terminate when **stagnation persists AND across 5 consecutive control-returns (not per-window — the taper makes returns sparse) the orchestrator intervened (a framework-rewrite OR a DR) yet could not break the stagnation.** Drop the "≥1 DR of 5" requirement.

**Behavioral problem today.** `interventions.jsonl` has **two disjoint row shapes** (rewrite-log vs work-score; `SKILL.md:388` vs `:111-124`) and `append_intervention` is never called by code — the agent writes rows by hand, so a control-return yields **0/1/2 rows** (H7). DR logs only to `calls.jsonl` (`deep_research.py:168-191`), so "≥1 DR" is **uncomputable** from `interventions.jsonl` (H6). There is no termination streak counter; "consecutive" + "involved an intervention" are eyeballed, a no-op return still writes a `work_score=0` row, and nothing defines what resets the streak (H8). The only real "consecutive" counter is `work_low_streak` (taper escalation) — a different concept that shares the magnitude 5, inviting conflation.

**Fix — one canonical control-return record + a deterministic, harness-computed predicate.**
- **Canonical row (exactly one per control-return):** `{type:"control_return", window_index, stagnation_flag, best_score, work_audit, work_dr, work_score, intervened:(work_audit>0||work_dr>0), outcome}`. It is simultaneously the taper input (`work_score`) and the termination input (`intervened` + `stagnation_flag` + `best_score`), eliminating the 0/1/2 split; folding DR into `work_dr` makes DR visible without a `calls.jsonl` join.
- **"Could not break stagnation"** (resolved at the *next* return, reusing the existing low-window bar from `stagnation_detector.stagnation_threshold`): the intervention at return *t* broke stagnation iff at *t+1* `stagnation_flag==False` **or** `best_score` gain > `max(stagnation_abs_floor, stagnation_rel_frac·max(best_score_t,0))`; else it failed.
- **Predicate (recommended owner: the HARNESS, a computed `return_reason="stagnation_intervention_exhausted"`, auto-finalized like `budget_exhausted`):** over the last 5 canonical rows, terminate iff each was stagnant, each had `intervened==True`, and each failed to break stagnation. **Reset** the streak on any non-intervention return or any stagnation-break. Evaluate at relaunch (the agent's row for the prior return already exists). Harness-computed is what the user's "simpler + reliable" asks for; agent-judgment is what produced H6/H8.
- **Coexistence:** stagnation already pins `target_cluster_size:1` (immediate return, `cadence_policy.py:53-54`), so the 5 stagnant returns actually accrue one-per-return instead of being smeared across a long taper cluster; a single healthy/taper return resets the count, so a quiet stretch can't accidentally terminate.
- **Files:** `journal.py` **[FND]** (canonical row + a `termination_streak` reader beside `work_low_streak` 391-403); `run_window.py` **[FND]** (`return_reason` block 1167-1217). **Reuse** the mutable `stagnation_detector.py` bar (don't hardcode). `cadence_policy.py` docstring note that taper-streak and termination-streak are now distinct.
- **Docs:** SKILL termination (439-446) + work-score (111-124) + diagnostics reasons (148-161) + journal readers (457-476); CLAUDE.md termination bullet (49-51); NOTES.md + the auto-memory wording carry the old "≥1 DR".
